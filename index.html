Jaka jest róźnica między OCR, a VL?
  1. OCR
    1. OCR to model/pipeline, którego głównym zadaniem jest odczytać tekst z obrazu dokumentu.
    2. Dobry OCR może też zwracać pozycje tekstu, bloki, tabele, czasem strukturę strony.
  1. VL / VLM to Vision-Language Model, czyli model obraz + język
    1. On nie tylko czyta tekst, ale też "rozumie" obraz jako całość i reaguje na prompt
    2. Czyli: "odczytaj tekst", ale też "zrób Markdown", "wyciągnij dane z faktury", "opisz tabelę", "znajdź podpis", "zmapuj pola do JSON".

Kiedy OCR, kiedy VLM, kiedy Markdown, kiedy LLM?

Główna zasada Nie każdy dokument wymaga ciężkiego modelu multimodalnego.
Tania ścieżka domyślna Jeśli dokument można sprowadzić do tekstu, najczęściej lepsza będzie ścieżka:
OCR / ekstrakcja tekstu → zwykły LLM → analiza
Kiedy VLM? VLM wybieramy wtedy, gdy znaczenie dokumentu zależy od obrazu, układu strony, elementów wizualnych albo lokalizacji obiektów.
Najtańsza dobra ścieżka Jeśli zwykły OCR i LLM wystarczą — nie ma sensu zaczynać od ciężkiego VLM.
Markdown ≠ OCR Model może dobrze czytać tekst, ale źle odtwarzać strukturę dokumentu. Dla Markdown testujemy strukturę osobno, nie tylko CER/WER.

Krok 1 — Czy to cyfrowy PDF z warstwą tekstową?

TAK Wyciągnij tekst klasycznie z PDF.
Użyj zwykłego LLM do klasyfikacji, ekstrakcji pól, streszczenia i pytań po polsku.
Nie zaczynaj od OCR / VLM.
NIE Przejdź do kolejnych kroków drzewka.
Dlaczego? PDF z tekstem daje natywną, doskonałą jakość ekstrakcji bez żadnego ryzyka błędów OCR czy halucynacji wizualnych. VLM i OCR-VLM są potrzebne dopiero tam, gdzie tekstu nie ma.

Kilka pojęć przed wynikami

Pojęcie Co oznacza
OCR Zwykłe odczytanie tekstu z obrazu lub skanu dokumentu.
VLM Model, który nie tylko czyta tekst, ale też "patrzy" na obraz: widzi układ strony, checkboxy, pieczątki, tabele, mapy i relacje przestrzenne.
CER Procent błędów na poziomie znaków. Im niżej, tym lepiej. Jeśli CER=0.0, model przepisał tekst idealnie.
WER Podobna metryka do CER, ale liczona na poziomie słów.
JSON Uporządkowany format odpowiedzi dla programu. Poprawny JSON nie oznacza jeszcze, że model zrozumiał dokument.
Markdown Tekst z zachowaną strukturą: nagłówki, listy, tabele. Przy dokumentach często ważniejszy niż surowy tekst.
bbox Prostokąt pokazujący, gdzie coś znajduje się na obrazie — np. pieczątka, pole formularza albo budynek na mapie.

GLM-OCR Szybki baseline — wygodny, ale nie ufaj samemu formatowi odpowiedzi
Qwen3-VL Najlepsze rozumienie obrazu i decyzji wizualnych
PaddleOCR-VL Najlepszy tani praktyczny OCR
HunyuanOCR Najlepszy sygnał w Markdown conversion + kandydat wielojęzyczny (wymaga potwierdzenia)
FireRed-OCR Najlepszy praktyczny wynik w Markdown conversion — ale ostrożnie przy decyzjach
Qianfan-OCR Kandydat do struktury dokumentu — wymaga większego runu
NuMarkdown ⚗ Ciekawy koncepcyjnie, obecny lokalny smoke nie potwierdził przewagi
GutenOCR-3B Lokalizacja i grounded OCR — gdzie ważne jest "gdzie", nie tylko "co"
DeepSeek-OCR 2 Wyspecjalizowany OCR — tekst i reading order, nie decyzje wizualne
dots ⚗ Ciekawy dla layoutu — technicznie niestabilny w obecnej integracji

Model może dobrze czytać tekst, ale źle rozumieć dokument.

Albo dobrze widzieć checkboxy, ale źle trzymać format odpowiedzi.

Dlatego benchmark musi mierzyć kilka różnych kompetencji, nie tylko jeden wynik OCR.
